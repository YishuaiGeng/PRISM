# PRISM 方法学审查：范式总结与成功/失败记忆检索是否合理、是否达到发表规范

> 2026-07-08。基于代码实现（prism/offline, prism/online, prism/paradigm_library）、
> 论文草稿（paper_draft_template_zh.tex）与本周 AR-LSAT 全流程实证（定标 42.5%、
> 半量消融、两轮离线挖掘试跑、带毒库事故）的交叉审查。
> 结论先行：**框架的核心主张（求解器作为经验准入门控）是合理且有区分度的，
> 但当前实现与论文叙述之间存在 6 处方法学缺口，其中 2 处（成功判据、跨题记忆
> 口径）如不修复/改写，会被认真的审稿人抓住并构成拒稿理由。**

---

## 一、成功记忆（范式总结）链条的逐环审查

流水线：轨迹收集 → KDP 识别 → 聚类 → LLM 抽象 → 三重筛选 → 入库。

### 1.1 【严重，已实证】轨迹"成功"判据与任务目标错位

- **现状**：`TrajectoryCollector` 的 solved = Z3-SAT + 模型域校验（`_solve_once`），
  **不校验答案正确性**。论文 §3.3 却写"仅保留最终 SAT 且答案正确的成功轨迹"——
  **代码与论文声明不一致**，这是规范性硬伤。
- **后果（已实证）**：ZebraLogic 上 SAT≈正确（约束完备时解唯一），错位被掩盖；
  AR-LSAT 上翻译错误常呈欠约束 SAT，挖出"改成相邻就 SAT"的伪范式
  （support 395/382 的两条 adjacent），注入后拖累端到端 −1.8pp。
- **修复**：① 本次新增 `verify_arlsat_trajectories.py`，用存储模型对五选项做
  三值判定（correct/incorrect/indeterminate，利用五选一良定性），
  `run_offline.py --trajectory-verdicts` 只从 correct 轨迹挖掘。
  **全量判定结果（2026-07-08，1,630 条）**：correct 122（7.5%）、
  incorrect 418（25.6%）、indeterminate 900（55.2%）、skipped 190。
  即**可判定的 SAT 轨迹中 77% 答案是错的**——"SAT ≠ 正确"在 AR-LSAT 上
  的量化证据，建议直接进论文（比任何论证都有力）。
  重挖后库：3 条（support 8/3/2），基底纯净但规模小；
  ② ZebraLogic 侧应同步把"答案与标准答案一致"写进 collector 或在论文中
  如实降格表述；③ 论文方法节必须明确定义 success criterion 并讨论
  "SAT ≠ 正确"的任务依赖性——这本身是对"验证门控"主张边界的有价值标定。

### 1.2 【中等】KDP→范式抽象缺少忠实性（faithfulness)验证

- **现状**：LLM 从 ≤10 个采样 KDP 归纳五元组。三重筛选检查的是**范式自身**
  的性质（自洽/非平凡/触发选择性），**不检查范式是否忠实概括了源聚类**——
  LLM 可以产出一条与聚类无关但恰好通过筛选的范式，现有机制无法发现。
- **审稿人问法**："how do you know the abstracted paradigm actually
  summarizes the cluster rather than being a plausible hallucination?"
- **修复（低成本）**：加"覆盖率回验"：抽象出的范式在源聚类 KDP 上的
  再触发率 / 操作模板匹配率，作为第四道筛选或至少作为报告指标。
  离线可算，不需要额外 LLM 调用。

### 1.3 【中等】聚类特征是"状态相似"的代理，不是"推理操作相似"

- **现状**：特征向量 = 约束类型直方图 + 域大小分布 + 步骤类型 one-hot + 规模。
  它刻画 KDP 发生时的**状态**，不刻画 KDP 执行的**操作**。两个状态相似、
  操作迥异的 KDP 会聚到一起——这是 AR-LSAT 试跑中 trigger 爆宽到 12 类的
  深层原因（已用"主导类型 + top-4 截断"缓解，但那是修剪症状）。
- **修复**：特征向量中加入操作签名维度（constraint_added 的算子结构：
  ==/!=/</Abs/Implies/Sum 的 one-hot），使聚类同时对齐状态与操作。
  改动小、可离线重聚类验证（轨迹已存盘，零 API 成本）。
  论文里需为特征设计给出 justification 或消融。

### 1.4 【中等】检索缺乏独立评测；触发精度筛选存在分布偏移

- **现状**：两层检索（类型包含 → LLM 语义判断）只有端到端消融
  （"仅用第一层检索 −1.9pp"），没有检索本身的 precision/recall 评测。
  且 τ_p 筛选在**固定 18 条 zebra 风格标注池**上测（本次已扩充
  implication/cardinality 条目），与在线真实约束分布不一致——
  在池上"选择性好"不代表在线不误触发。
- **审稿人问法**："retrieval quality is claimed but never measured."
- **修复**：抽 100 个在线状态，人工/LLM 标注每条范式是否适用，报告两层
  检索的 P/R——一天工作量，能把 §范式检索从"设计描述"升级为"有测量的组件"。
  另：AR-LSAT 上 Layer-2 使调用数近乎翻倍（2.97→5.1+）却无准确率收益，
  成本-收益需要在论文中如实报告。

## 二、失败记忆链条的审查

### 2.1 【严重-中等】错误范式的表示层次与论文主张之间有 gap

- **论文主张**：LLM 存在"跨题目、系统性"的翻译失败模式（方向翻转、
  过度对称化…），可通过 UNSAT core 指纹自动识别并建立规避模板。
- **实现现状**：错误范式的去重键是 **UNSAT core 的 SHA-256 指纹**——
  这是**实例级**表示：同一类语义错误在不同题目上 core 文本不同 → 指纹不同
  → 无法聚合。全量挖掘 131 条、平均 support 2.83、大量 support=1 的
  一次性签名，正说明指纹难以跨题聚合。"系统性失败模式"在当前表示下
  **推导不出来**，scope 里的约束类型分布只是侧面统计。
- **端到端佐证**：带错误库的"仅范式库"两版（131 条 / 51 条）都没有
  相对基线的正增益——错误规避的价值目前**没有实证支撑**。
- **修复方向**（二选一，影响论文写法）：
  a) 把错误范式的键从 core 指纹升级为**类型化错误签名**
     （error_type × 约束类型对 × 算子模式，如 SEMANTIC_FLIP × directly_left
     × 符号反转），指纹只做题内停滞检测——表示分层：题内用指纹、
     跨题用类型签名。这与草稿 §3.4 的 ET-001 描述其实更接近；
  b) 若不改实现，论文必须把"系统性失败模式"降格为对 zebra 的观察
     （E-003/E-004 那批是 zebra 挖的），AR-LSAT 侧只 claim 机制可运行。
- **注意**：草稿表 5（19 条错误范式、E-002~E-007 分类）来自旧 zebra 库，
  与 AR-LSAT 全量挖掘的 131 条形态不同，投稿版必须统一口径。

### 2.2 【严重（写作层面）】"修复轨迹记忆"是题内机制，跨题主张与证据错位

- **现状**：`RepairMemory` 在 `GuidedSolver.solve()` 里**每题重新初始化**
  （`memory = RepairMemory(...)` per puzzle）。它做的是题内停滞检测
  （Jaccard）、循环检测、策略切换。**跨实例**的部分只有写回机制
  （消融显示 −1.4pp，贡献很小）。
- **摘要现状**："不同问题实例相互独立求解，难以积累和复用跨实例经验……
  类型化修复轨迹记忆基于 UNSAT core 归因记录……"——读者（和审稿人）
  会把 +3pp/停滞率下降归因于"跨实例记忆"，而机制上它来自**题内**
  停滞检测与策略升级。这是 claims-evidence alignment 问题，属于
  审稿人最反感的一类（overclaiming by ambiguity）。
- **修复（纯写作，必须做）**：术语分家——
  "repair-trajectory memory (within-instance)" 与
  "experience write-back (cross-instance)" 明确二分；摘要与贡献 3 的
  表述相应改写。跨实例主张只挂在范式库 + 错误库 + 写回上，
  并用对应消融数字支撑。

## 三、统计与实验规范

- 【必须】所有主对比 3 种子 + 配对 bootstrap；当前 AR-LSAT 半量为单种子
  （仅修复记忆 +3.0pp, p=0.070——方向对但未达标）。
- 【必须】同协议原则：fallback、修复预算 R、种子、基础模型全表统一
  （已做：--fallback random 默认）。
- 【建议】报告"提取率/歧义率/回退占比"三件套，防止"准确率提升来自
  回退运气"的质疑（数据已在 CSV 里）。
- 【建议】范式库规模、触发率、命中率按 benchmark 分开报告；
  AR-LSAT 上 Layer-2 的调用开销单独列。

## 四、总体判断

| 维度 | 判断 |
|---|---|
| 核心主张（验证门控记忆准入） | 合理、有区分度，且本周的"带毒库→curation"事故给了它一个诚实的边界标定 |
| 成功记忆链条 | 骨架合理；1.1 必须修（进行中），1.2/1.3 修复成本低收益明显 |
| 失败记忆链条 | 题内部分（停滞检测+策略切换）是目前唯一有实证增益的机制（+3pp）；跨题部分（错误范式）表示层次不支撑论文主张，需改实现或改口径 |
| 发表规范水准 | 当前**未达到**：代码-论文不一致（1.1）、机制归因含混（2.2）是硬伤；补齐 1.1/2.2 + 3 种子后达到可辩护水准 |
